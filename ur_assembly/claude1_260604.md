완벽한 가이드를 만들기 위해, 아직 다루지 않은 **실제 로봇 배포(Isaac ROS UR DNN policy)** 부분과 **설치/UR16e 에셋** 정보를 정확히 확인하겠습니다.필요한 정보를 모두 확인했습니다. 핵심 사실 몇 가지만 짚고, 전체 가이드는 따라 하며 저장할 수 있게 파일로 정리해 드리겠습니다.

확인된 중요 사실: gear assembly task는 Isaac Lab에서 PPO로 학습되어 UR10e에 UR torque interface로 배포됐고, RTX 4090 GPU에서 학습된 zero-shot sim-to-real입니다. UR의 직접 토크 제어 인터페이스가 impedance control을 가능하게 해 안전하고 compliant한 접촉을 만듭니다. 실제 배포는 `isaac_manipulator_ur_dnn_policy` 패키지의 inference.launch.py로 실행하며, Segment Anything으로 객체를 세그먼트(사용자가 gear base 중앙을 클릭해 힌트)한 뒤 pose를 추정합니다. UR16e는 Isaac Sim 매니퓰레이터 튜토리얼에서 ur_description으로 UR 로봇을 URDF 임포트(ur_type 지정)하고 Gain Tuner로 physics·joint effort limit·gain을 설정(critically damped: Nat. Freq. 0.5, Damping Ratio 1.0)하는 동일 절차로 셋업합니다.

이제 UR16e 기준으로 재구성한 완전한 가이드입니다.UR16e 기준으로 재구성한 완전한 가이드를 파일로 만들었습니다. 검증된 `Isaac-Deploy-GearAssembly-UR10e-2F140-v0` 워크플로우를 사용자 task로 이식하는 전체 경로를 담았습니다.

가이드의 핵심 흐름과 주의점만 짚으면:

**가장 중요한 3가지 (UR16e 전환의 실패 포인트)**
1. **Actuator 스펙 교체** — UR10e 기본값(effort_limit=87.0 등)을 그대로 쓰면 안 되고, UR16e 데이터시트의 관절 토크 한계로 바꿔야 합니다. 16kg 페이로드라 토크 사양이 다릅니다.
2. **Stiction 실측·모델링** — UR16e에 impedance 컨트롤러를 걸고 step response를 측정해 정상상태 오차를 확인한 뒤, 그 마찰을 sim에 추가하는 작업이 sim-to-real 정확도를 좌우합니다.
3. **Physics 캘리브레이션** — wheel/shaft 재질 마찰계수를 실하드웨어 영상과 시각 비교로 맞추는 반복 작업.

**학습·배포 요약**
- 학습: PPO(RSL-RL), 256 envs, RTX 5090에서 12~24시간. solver iteration은 4~16 유지(더 높이면 40GB+ 필요).
- 배포: Isaac ROS `inference.launch.py` + RealSense + Segment Anything(클릭 힌트) + FoundationPose + UR torque interface.

**볼팅/언볼팅**은 별도 스킬로 분리하는 걸 권합니다. insertion보다 본질적으로 어렵고(cross-threading/jamming), 회전+토크 제어가 필요해서, wheel-shaft insertion을 먼저 성공시킨 뒤 단계적으로 확장하는 마일스톤을 9장에 정리했습니다.

한 가지 솔직히 덧붙이면, 가이드의 task 이름(`Isaac-Deploy-WheelShaft-UR16e-2F140-v0`)은 **사용자가 직접 만들 환경의 예시 이름**입니다. 원본 gear assembly env를 복제해서 로봇/에셋/observation을 교체하는 식으로 새 task를 등록하게 됩니다.

## 2. UR16e 로봇 셋업 (UR10e → UR16e 전환의 핵심)

검증된 task는 UR10e 기준입니다. UR16e로 바꾸려면 아래를 수행합니다.

### 2.1 UR16e USD 준비 (URDF 임포트)
Isaac Sim 매니퓰레이터 튜토리얼 절차를 그대로 따르되 로봇 타입만 ur16e로 지정:
```bash
# ur_description로 UR16e 확인 (ROS2)
ros2 launch ur_description view_ur.launch.py ur_type:=ur16e
```
- Isaac Sim에서 ROS2 URDF Node → robot_state_publisher로 임포트
- 조인트 설정 시 Natural Frequency 적용 (튜토리얼 권장값 참고)
- 참고: https://docs.isaacsim.omniverse.nvidia.com/latest/robot_setup_tutorials/tutorial_import_assemble_manipulator.html

### 2.2 Physics / Gain 설정 (Gain Tuner)
- Tools > Robotics > Asset Editors > Gain Tuner
- critically damped 기준: **Natural Frequency = 0.5, Damping Ratio = 1.0** (튜토리얼 권장)
- 그리퍼 핑거에 physics material 추가(안정적 grasp)
- 참고: https://docs.isaacsim.omniverse.nvidia.com/latest/robot_setup_tutorials/tutorial_configure_manipulator.html

### 2.3 Actuator 설정 — UR16e 스펙으로 교체 (★중요)
gear assembly의 UR10e 기본 actuator 설정 예시:
```python
# UR10e 기본값 (참고용 — 그대로 쓰면 안 됨)
actuators = {
    "arm": ImplicitActuatorCfg(
        joint_names_expr=["shoulder_pan_joint", "shoulder_lift_joint",
                          "elbow_joint", "wrist_1_joint", "wrist_2_joint", "wrist_3_joint"],
        effort_limit=87.0,     # ← UR10e 값. UR16e 데이터시트 값으로 교체
        velocity_limit=2.0,    # ← UR16e 관절 속도 한계로 교체
        stiffness=800.0,       # ← 실제 거동에 맞춰 재캘리브레이션
        damping=40.0,          # ← 재캘리브레이션
    ),
}
```
**해야 할 일**: UR16e는 페이로드 16kg으로 토크 사양이 다릅니다. `effort_limit`을 UR16e 데이터시트의 관절별 토크 한계로 바꾸고, `stiffness/damping`은 아래 2.4의 step‑response 측정으로 재조정하세요. UR10e 값(87.0 등)을 그대로 쓰면 sim‑real gap이 커집니다.

### 2.4 관절 마찰/Stiction 모델링 (sim‑real 정확도의 핵심)
실측 절차:
1. UR16e에 impedance 컨트롤러를 걸고 step response 측정
2. 명령 setpoint 대비 정상상태 오차(각도) 확인 (UR10e 사례는 약 0.25°)
3. 그 오차를 만드는 관절 마찰을 sim에 추가
```python
joint_friction = EventTerm(
    func=mdp.randomize_joint_parameters,
    mode="reset",
    params={
        "asset_cfg": SceneEntityCfg("robot", joint_names=["shoulder_.*", "elbow_.*", "wrist_.*"]),
        "friction_distribution_params": (0.3, 0.7),  # UR16e 실측에 맞춰 조정 (Nm)
        "operation": "add",
        "distribution": "uniform",
    },
)
```
> 마찰을 모델링하지 않으면 정책이 "명령한 관절 위치가 항상 도달된다"고 잘못 학습합니다. 실로봇의 stiction이 작은 움직임을 막아 정상상태 오차가 생기므로, sim에 마찰을 넣어야 정책이 충분히 큰 모션을 명령하도록 배웁니다. 필요시 action scale도 약간 키워 stiction을 극복(예: 0.025 → 0.0325).

## 4. RL 환경 정의 (Observation / Action / Reward / Randomization)

### 4.1 Observation (F/T 없음)
| 관측 | 차원 | 실로봇 소스 | 학습 시 노이즈 |
|---|---|---|---|
| joint_pos | 6 | UR16e 컨트롤러 | 없음(proprioceptive) |
| joint_vel | 6 | UR16e 컨트롤러 | 없음 |
| target_pos | 3 | FoundationPose + RealSense depth | ±5mm |
| target_quat | 4 | FoundationPose + RealSense depth | ±~5° |

```python
@configclass
class PolicyCfg(ObsGroup):
    joint_pos = ObsTerm(func=mdp.joint_pos, params={"asset_cfg": SceneEntityCfg("robot", joint_names=[...])})
    joint_vel = ObsTerm(func=mdp.joint_vel, params={"asset_cfg": SceneEntityCfg("robot", joint_names=[...])})
    # 비전 기반 관측에만 노이즈 추가 (perception 오차 모사)
    target_pos = ObsTerm(func=mdp.target_pos_w, params={"asset_cfg": SceneEntityCfg("shaft_base")},
                         noise=Unoise(n_min=-0.005, n_max=0.005))
    target_quat = ObsTerm(func=mdp.target_quat_w, params={"asset_cfg": SceneEntityCfg("shaft_base")},
                          noise=Unoise(n_min=-0.01, n_max=0.01))
    def __post_init__(self):
        self.enable_corruption = True   # perception 관측만 corruption
        self.concatenate_terms = True
```
> proprioceptive 관측(joint_pos/vel)에는 노이즈를 넣지 않는 것이 전이에 더 좋다는 것이 경험적 결과입니다(UR 컨트롤러의 관절 피드백이 충분히 정확).

### 4.2 Action — incremental joint position
```python
self.joint_action_scale = 0.025  # 스텝당 ±2.5도 (contact-rich는 작게)
self.actions.arm_action = mdp.RelativeJointPositionActionCfg(
    asset_name="robot",
    joint_names=["shoulder_pan_joint","shoulder_lift_joint","elbow_joint",
                 "wrist_1_joint","wrist_2_joint","wrist_3_joint"],
    scale=self.joint_action_scale,
    use_zero_offset=True,
)
```
- action scale은 task 정밀도(작을수록 정밀)와 제어 주파수(높을수록 큰 스텝 허용)로 튜닝

### 4.3 System Response Consistency (physics 캘리브레이션)
마찰계수 캘리브레이션 절차(반복):
1. 실하드웨어에서 부품 grasp/조작 영상 녹화
2. sim viewer 관찰 → penetration·비현실적 미끄러짐·접촉 불량 확인
3. 마찰/solver 조정 후 재시도 → 거동이 일치할 때까지 반복(전체 학습 기다릴 필요 없음)
```python
small_part_physics_material = EventTerm(
    func=mdp.randomize_rigid_body_material, mode="startup",
    params={
        "asset_cfg": SceneEntityCfg("wheel", body_names=".*"),
        "static_friction_range": (0.75, 0.75),   # 실제 재질에 맞춰
        "dynamic_friction_range": (0.75, 0.75),
        "restitution_range": (0.0, 0.0),
        "num_buckets": 16,
    },
)
```
Rigid body / solver 설정:
```python
rigid_props=sim_utils.RigidBodyPropertiesCfg(
    disable_gravity=True,                 # 로봇은 고정 마운트
    max_depenetration_velocity=5.0,
    enable_gyroscopic_forces=True,
    solver_position_iteration_count=4,    # contact-rich 핵심. penetration 시 8/16으로
    solver_velocity_iteration_count=1,
    max_contact_impulse=1e32,
)
```

### 4.4 Domain Randomization
```python
# actuator gain randomization
robot_joint_stiffness_and_damping = EventTerm(
    func=mdp.randomize_actuator_gains, mode="reset",
    params={
        "asset_cfg": SceneEntityCfg("robot", joint_names=["shoulder_.*","elbow_.*","wrist_.*"]),
        "stiffness_distribution_params": (0.75, 1.5),   # 75~150%
        "damping_distribution_params": (0.3, 3.0),      # 30~300%
        "operation": "scale", "distribution": "log_uniform",
    },
)
# 부품/베이스 pose randomization (워크스페이스 커버)
randomize_parts_and_base_pose = EventTerm(
    func=...,
    mode="reset",
    params={"pose_range": {"x":[-0.1,0.1], "y":[-0.25,0.25], "z":[-0.1,0.1],
                           "yaw":[-math.pi/6, math.pi/6], ...}},
)
```
> 원칙: 실로봇이 동작해야 할 범위를 커버하되, 너무 넓히면 학습이 어려워짐. 학습 난이도와 robustness의 균형.

### 4.5 Reward (직접 설계 필요)
gear assembly는 insertion 보상을 사용합니다. 일반적 구성:
- 타겟까지 거리/keypoint 정렬 보상(연속)
- 삽입 깊이 도달 보상
- 성공 보너스 + 과도한 접촉/이탈 페널티
- 볼팅의 경우: 회전 진행도(나사산 결합)와 토크 한계 페널티를 추가
> 보상 설계가 이 방식의 가장 어려운 부분입니다. gear assembly env의 reward term을 베이스로 시작해 점진적으로 shaping.

reward 증가 추세로 학습 정상 여부 확인.

## 6. 실제 UR16e 배포 (Isaac ROS)

### 6.1 배포 입력
학습 산출물(`.pt` 체크포인트 + `agent.yaml` + `env.yaml`)을 Isaac ROS 추론 파이프라인이 직접 사용. 별도 export 불필요.

### 6.2 Perception 셋업
1. 카메라 캘리브레이션(예: `<calib_name>` 지정)
2. rqt_image_view에서 **타겟 베이스 중앙을 클릭**해 Segment Anything에 세그먼트 힌트 제공
   - 그리퍼가 타겟을 가리지 않게 장면에서 제거/회피
   - 세그먼트 토픽: `/segment_anything/colored_segmentation_mask`
3. FoundationPose가 pose 추정 → 정책 관측으로 전달

### 6.3 추론 실행
```bash
ros2 launch isaac_manipulator_ur_dnn_policy inference.launch.py \
    checkpoint:=<WHEEL_SHAFT_MODEL_FILE_NAME>
```
파이프라인 구성: Perception(SAM + FoundationPose) → Motion Planning(cuMotion) → Policy Inference(ROS 노드) → Robot Control(UR torque interface, impedance)

### 6.4 sim‑to‑real 디버깅 원칙
실패 시: 실로봇을 sim과 동일한 초기 관측으로 세팅 → 컨트롤러/시스템 반응 비교. 이를 통해 문제가 **관측 불일치(Input Consistency)**인지 **physics/controller 불일치(System Response)**인지 분리.

3대 일관성 체크리스트:
- Input: sim 관측이 실센서로 얻을 수 있는 것만 사용(privileged 정보 금지)
- System Response: physics(마찰/접촉) + actuator(gain/limit/마찰) 일치
- Output: 정책 출력 후처리(action scale 등)가 sim/real 동일

## 7. 볼팅/언볼팅 특화 고려사항

볼팅은 insertion과 다른 스킬로 분리하세요.
- **난이도**: 나사 조립은 cross‑threading/jamming 등 본질적 실패 모드 존재 → insertion보다 성공률 낮음
- **레퍼런스**: FORGE의 nut threading + GearMesh 환경(Franka 기준이라 UR 이식 필요)
- **제어**: 회전 + 토크 한계 제어가 핵심. UR torque interface 활용
- **언볼팅**: 볼팅의 역방향(풀기). 초기 그립/정렬 + 역회전. 볼트 머리/너트 pose 추정 정확도가 관건
- **권장 순서**: 1단계 wheel‑shaft insertion 성공 → 2단계 wheel 제거 → 3단계 볼팅 → 4단계 언볼팅

## 8. Troubleshooting

### PhysX Collision Stack Overflow
```
PxGpuDynamicsMemoryConfig::collisionStackSize buffer overflow detected ...
```
→ 환경 cfg에서 버퍼 증가:
```python
sim: SimulationCfg = SimulationCfg(
    physx=PhysxCfg(
        gpu_collision_stack_size=2**31,   # 에러 메시지 권장값 이상으로
        gpu_max_rigid_contact_count=2**23,
        gpu_max_rigid_patch_count=2**23,
    ),
)
```

### CUDA Out of Memory (RTX 5090에서 발생 시)
우선순위 순:
1. `--num_envs` 줄이기 (256 → 128/64) — 샘플 다양성↓, 수렴 느려질 수 있으나 최종 품질은 유사
2. solver_position_iteration_count를 4로 되돌리기 (높였을 경우)
3. `--video` 플래그 제거

### 핵심 요약
- 검증된 UR sim‑to‑real 경로(gear assembly)를 wheel‑shaft로 이식하는 것이 가장 빠른 성공 경로
- UR16e 전환의 핵심은 **actuator 스펙 교체 + stiction 실측·모델링 + physics 캘리브레이션**
- 학습은 PPO/RSL‑RL, RTX 5090에서 256 envs로 충분, 배포는 Isaac ROS + UR torque interface
- 볼팅/언볼팅은 별도 스킬로 분리하고 insertion 성공 후 단계적으로 확장
- F/T는 이 경로에서 필수 아님(후속 옵션)
